Introduction

Purpose
This treatment plan documents the risk treatment strategies selected for five cloud infrastructure assets identified during the risk assessment phase. The purpose is to systematically address the identified risks using one of four treatment approaches — Avoid, Mitigate (Control), Transfer (Share), or Accept (Retain) — and to provide a structured plan for implementing the chosen strategy, including action items, timelines, risk ownership, and residual risk assessment.

This document aligns with ISO/IEC 27001:2022 Clause 6.1.3 (risk treatment), ISO 31000:2018, and the AWS Well-Architected Framework Security Pillar. Treatment strategies are also cross-referenced with CSA Cloud Controls Matrix (CCM) v4 where applicable.
Scope
This treatment plan covers five AWS cloud infrastructure assets with identified risks ranging from Critical to Medium. The treatment plans focus on risk treatment approach and rationale — the specific technical implementation mechanisms will be addressed in the subsequent security hardening lab. Assets in scope:
  • Asset 1 — AWS Managed Microsoft AD CRITICAL Two-way trust with unpatched legacy domain controller; shared Domain Admin account
  • Asset 2 — Amazon RDS HIGH Public accessibility enabled; no encryption at rest; public subnet with no NACLs
  • Asset 3 — Amazon EC2 Instances HIGH End-of-life OS; Security Groups permitting all inbound on SSH/RDP from 0.0.0.0/0
  • Asset 4 — Amazon EFS MEDIUM No File System Policy; IAM authentication not enforced; encryption-in-transit not mandated
  • Asset 5 — AWS Lambda Functions MEDIUM AdministratorAccess policy attached to execution roles far exceeding task requirements
⚠️
Critical vs Medium Prioritisation — Key Principle
Critical and High risks (Assets 1–3) must be addressed before Medium risks (Assets 4–5) because they represent immediate existential threats to the organisation — total environment compromise, regulatory breach, and active exploitation via internet-exposed ports. Medium risks, while important, do not pose the same immediate danger and can be addressed on a scheduled timeline after Critical/High remediation is complete. See the Collaborative Strategy tab for the full prioritisation rationale.

Identified Residual Risks

  • Risk 1 — Legacy Trust Chain Compromise: The two-way trust relationship with an unpatched, legacy on-premises domain controller creates a bidirectional attack path. If the legacy controller is compromised (highly likely given it is unpatched), the attacker gains transitive access to the cloud-based Managed AD, and from there, to all resources in the domain — including production databases, EC2 instances, and file systems.
  • Risk 2 — Credential Compromise via Shared Admin Account: All administrative actions using a single shared "Domain Admin" account means that a single stolen credential grants unrestricted administrative access to the entire AD environment. There is no way to attribute specific actions to individual administrators, making forensic investigation impossible and audit trails meaningless.

Treatment Plans

🚫
Risk 1 Treatment: AVOID — Remove Two-Way Trust Relationship
ISO 27001 A.5.19 Information Security in Supplier Relationships | CSA CCM IAM-02
Justification: The Avoid strategy is the most appropriate treatment for the legacy trust relationship because the risk cannot be adequately mitigated while the trust exists — the unpatched legacy domain controller will remain a critical vulnerability regardless of compensating controls. Avoidance by removing the two-way trust eliminates the attack path entirely. The business need for the trust should be formally re-evaluated; if connectivity between on-premises and cloud identity is genuinely required, a one-way, read-only trust with a fully patched domain controller should replace it.

Identified Residual Risks

  • Risk 1 — Internet-Exposed Database: Public accessibility enabled with no restrictive NACLs means the RDS endpoint is directly reachable from the internet. Combined with likely default or weak credentials, this creates an Almost Certain likelihood of unauthorised access, data exfiltration, or ransomware deployment against production database data.
  • Risk 2 — Unencrypted Data at Rest: Encryption at rest is not enabled via AWS KMS. If the underlying storage media is ever compromised — through snapshot exposure, storage volume reuse, or insider access — all production data is readable in plaintext without any additional decryption requirement.

Treatment Plan

MITIGATE — Remove Public Access + Encrypt at Rest + Restrict Network
ISO 27001 A.8.24 Cryptography | A.8.20 Network Security | A.8.22 Network Segregation
Justification: Mitigation is the appropriate strategy for RDS because the database is a core operational asset that cannot be avoided or transferred without significantly disrupting the application. Acceptance is not appropriate at High risk level for a production database holding city service data. Mitigation addresses all three root causes: public accessibility (network control), unencrypted data (cryptographic control), and permissive NACLs (network segmentation). Transfer (insurance) may be considered as a supplementary measure for residual financial exposure but does not address the technical vulnerability.

🟠 HIGH RISK Strategy: MITIGATE + AVOID Asset 3

Amazon EC2 Instances

Identified Residual Risks

  • Risk 1 — End-of-Life Operating Systems: EC2 instances running EoL OS (e.g. Windows Server 2012, Ubuntu 18.04) receive no vendor security patches. Known CVEs accumulate without remediation, providing attackers with a permanent, growing catalogue of exploitable vulnerabilities. This cannot be mitigated by network controls alone — the underlying OS is permanently vulnerable.
  • Risk 2 — Internet-Exposed SSH and RDP: Security Groups permitting all inbound traffic from 0.0.0.0/0 on ports 22 and 3389 expose the instances to continuous automated brute-force attacks. Per NCSC honeypot data, exposed RDP without MFA is typically attacked within 24 hours of deployment.
3

Treatment Plans

🚫
Risk 1 Treatment: AVOID — Decommission End-of-Life OS Instances
ISO 27001 A.8.8 — Management of Technical Vulnerabilities | CSA CCM TVM-01
Justification: Avoidance (decommissioning and replacing EoL instances) is the only technically sound strategy for end-of-life operating systems. Mitigation via compensating controls (additional firewall rules, WAF) can reduce exposure but cannot patch the underlying unpatched kernel and application vulnerabilities — the risk source itself remains. Transfer (insurance) does not address the technical vulnerability. Acceptance is inappropriate at High risk level. The instances must be replaced with supported OS versions on new AMIs. If the application workload cannot be migrated immediately, an Extended Security Updates (ESU) programme should be evaluated as a temporary bridge measure only.
ISO 27001 A.8.8 — Technical Vulnerability Management CSA CCM TVM-01AWS WAF SEC09 — Incident Response NCSC Vulnerability Management Guidance
🛡️
Risk 2 Treatment: MITIGATE — Restrict SSH/RDP + Eliminate Internet Exposure
ISO 27001 A.8.20 Network Security | A.8.5 Secure Authentication | CSA CCM IAM-01
Justification: Mitigation is the correct treatment for exposed SSH/RDP — the ports cannot be simply avoided (remote management access is operationally necessary) but the exposure can be dramatically reduced. Removing 0.0.0.0/0 rules and replacing direct SSH/RDP with AWS Systems Manager Session Manager (which requires no inbound ports) eliminates the internet-facing attack surface entirely while preserving administrative capability. This is the AWS Well-Architected Framework recommended approach for EC2 remote access.

Amazon Elastic File System (EFS)

No File System Policy · Any VPC resource can mount · Encryption-in-transit not mandated

Identified Residual Risks

  • Risk 1 — Unrestricted VPC Mount Access: Without a File System Policy enforcing IAM authentication, any EC2 instance, Lambda function, or ECS container within the VPC can mount the EFS volume and read or write its contents. A compromised workload in the VPC (e.g. a Lambda function or EC2 instance) could access all EFS data without any additional authentication.
  • Risk 2 — Unencrypted Data in Transit: Without mandatory encryption-in-transit at mount targets, data moving between EFS and EC2 instances over the internal VPC network is transmitted in plaintext. While internal network interception is less common than internet-based attacks, it is exploitable in lateral movement scenarios following an initial compromise.
3

Treatment Plan

🛡️
MITIGATE — Enforce IAM Authentication + Mandatory Encryption-in-Transit
ISO 27001 A.8.3 Access Restriction | A.8.24 Cryptography | A.5.15 Access Control
Justification: Mitigation is the correct and sufficient treatment for Medium-risk EFS vulnerabilities. The risks are specific, well-bounded, and addressable through configuration changes without operational disruption. Avoidance (not using EFS) is disproportionate. Acceptance is not appropriate where a straightforward technical fix exists. Mitigation via a resource-based File System Policy enforcing IAM authentication, combined with mandatory TLS enforcement at mount targets, eliminates both identified risks without affecting legitimate workload access.
ISO 27001 A.8.3 — Information Access Restriction A.8.24 — Use of Cryptography A.5.15 — Access Control CSA CCM DSP-03 AWS WAF SEC05 — Protect Networks AWS WAF SEC06 — Protect Data

Identify all current EFS clients (EC2 instances, Lambda functions, ECS tasks) that legitimately mount the file system — document which IAM roles each usesShort-termWithin 1 weekCloud Security Lead

Add a condition to the File System Policy requiring "aws:SecureTransport": true — this mandates TLS for all mount connections, rejecting unencrypted mountsShort-termWithin 2 weeksCloud Security Lead

After enforcing IAM authentication and mandatory TLS: unauthorised mounts are blocked at the policy level, and transit data is encrypted. Residual risk: a compromised workload with legitimate EFS IAM permissions could still access the data — this is an inherent residual risk of shared file systems. Score: Low. Accepted by Cloud Security Lead with quarterly IAM permission audit for EFS client roles.

Identified Residual Risks

  • Risk 1 — Excessive Permissions (Privilege Escalation Path): Lambda execution roles with AdministratorAccess can modify IAM policies, delete S3 buckets, terminate EC2 instances, and perform any other AWS action. If a Lambda function is compromised through code injection, event data manipulation, or a dependency vulnerability, the attacker immediately has full AWS account administrator access — the most severe possible blast radius in an AWS environment.
  • Risk 2 — Violation of Least-Privilege Principle: Assigning AdministratorAccess to data processing functions violates a foundational security principle (ISO 27001 A.8.2, AWS WAF SEC03). Even without an active exploit, this configuration creates audit and compliance risk — demonstrating to ISO 27001 auditors or ICO investigators that appropriate access controls were not in place for functions handling personal or operational data.
3

Treatment Plan

🛡️
MITIGATE — Replace AdministratorAccess with Least-Privilege Custom Policies
ISO 27001 A.8.2 Privileged Access | A.5.15 Access Control | CSA CCM IAM-05
Justification: Mitigation via least-privilege policy replacement is the clear and correct treatment for excessive Lambda permissions. Avoidance (not using Lambda) is operationally disruptive and disproportionate to the risk level (Medium). Acceptance violates the least-privilege principle and would be flagged in any ISO 27001 audit. Transfer does not address the technical vulnerability. Mitigation — creating task-specific IAM policies granting only the exact permissions each function needs — directly eliminates the over-permissioned risk without any operational impact to the functions themselves, since they only need the permissions they actually use.

PRIORITY

Why Critical and High Risks Must Be Treated First

The following rationale justifies the prioritisation of Assets 1–3 (Critical/High) over Assets 4–5 (Medium) in the treatment timeline:
Immediate — Week 1
Asset 1: Managed AD (CRITICAL) — Trust Removal + Admin Account Elimination
A CRITICAL risk represents a threat of total environment compromise. If the legacy trust chain is exploited, all other assets become accessible regardless of their individual controls. The Managed AD compromise is the "master key" risk — it must be treated first because it undermines all other security measures. The shared Domain Admin account means a single stolen credential grants unrestricted control over the entire domain. These risks are actively exploitable with tools already deployed against internet-exposed infrastructure.
Short-term — Weeks 1–2
Assets 2 & 3: RDS + EC2 (HIGH) — Remove Internet Exposure
HIGH risks represent active, immediate threats with serious consequences. The internet-exposed RDS endpoint and the 0.0.0.0/0 SSH/RDP Security Groups are being continuously scanned and attacked right now — not theoretically. NCSC data confirms that exposed RDP without MFA is attacked within 24 hours. EoL OS instances have accumulating unpatched CVEs. These risks must be treated urgently because their likelihood of exploitation is very high and the consequence (data breach, ransomware, account takeover) is severe and directly impacts the city government partnership. The initial mitigations (removing 0.0.0.0/0 rules, disabling RDS public access) can be completed within 24–48 hours with minimal operational disruption.
Medium-term — Weeks 3–4
Assets 4 & 5: EFS + Lambda (MEDIUM) — Configuration Hardening
MEDIUM risks are important but do not represent immediate or near-certain exploitation. The EFS risk requires an attacker to already be inside the VPC with a compromised workload — it is a consequence-amplifier for a prior compromise, not an initial access vector. The Lambda over-permission risk requires an attacker to compromise a specific Lambda function's execution context. Both risks are real and must be treated, but they are lower probability in the near term and can be addressed once the higher-priority risks that create the conditions for their exploitation (network exposure, identity compromise) have been mitigated. Treating Medium risks while Critical/High risks remain unaddressed is resource misallocation.

Risk Mitigation — Contingency Plans

Contingency Plans
  • Asset 1 — Trust removal breaks dependent application: Suspend rather than delete the trust immediately; maintain a rollback capability for 48 hours. Engage application owner to identify and replace the trust dependency before permanent removal. Temporary workaround: application-specific service accounts with narrowly scoped credentials.
  • Asset 2 — RDS migration causes application connectivity failure: Maintain a read-only snapshot of the public-access instance for 72 hours after migration. Developer bastion host is available as a fallback for legitimate remote DB access if SSM access cannot be configured in time.
  • Asset 3 — EC2 OS migration breaks application workload: All OS migrations must have a tested rollback AMI. Production migration must follow staging environment validation. Migrations scheduled during maintenance windows with city government consent to minimise service impact.
  • Asset 4 — EFS policy blocks legitimate workload: The File System Policy is additive — test in audit mode before enforcing. Maintain a list of all legitimate EFS client IAM roles and verify permissions before enforcement.
  • Asset 5 — Lambda least-privilege policy causes function failure: Staging environment validation is mandatory before production deployment. Rollback plan: maintain the original AdministratorAccess role attachment as a disabled version for 30 days in case rapid rollback is needed.
Risk Reassessment
A formal risk reassessment (Report 2) will be conducted within two weeks of completing all treatment actions. The reassessment will: (1) re-score all 15 risks against the same 5-point Likelihood × Consequence scale; (2) document the reduction in risk scores as evidence of treatment effectiveness; (3) formally accept any residual risks above the acceptance threshold (Medium ≤6) with risk owner sign-off; (4) identify any new risks introduced by the treatment actions themselves (e.g. SSM as a new attack surface, new IAM roles created for individual admin access). The reassessment output becomes Report 2 in the RiskAnalysis.docx template, with Section 7 (Risk Treatment) fully populated.
